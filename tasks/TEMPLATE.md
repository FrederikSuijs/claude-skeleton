## Context

Why this work matters. 1–3 sentences. Link to issue / ticket / design doc
if one exists. State any non-obvious constraints the agent should know
up front (perf budget, compatibility floor, blast radius, etc.).

## Goal

One sentence. What does "done" look like in observable terms?

## Acceptance criteria

- [ ] ...
- [ ] ...
- [ ] ...

Each criterion must be **checkable**: a test, a CLI invocation, a
screenshot, a benchmark number, a `git grep` that should return zero
hits. If you can't say how to verify it, the criterion isn't concrete
enough.

## Out of scope

- ... (things a future task will own)

## Approach (optional)

A short sketch of the implementation. Skip this section if the approach
is obvious from the goal. The point is to lock in tradeoffs *before* the
agent picks one — not to write the code for them.

## Files likely touched

- `path/to/file.ext` — what changes and why
- `path/to/other.ext` — what changes and why

## Verify

The exact commands to run, in order, that prove the acceptance criteria
are met. The agent should run these and report results, not paraphrase.

```bash
# example
make lint
make test
```

## Notes

Anything else: links, prior art, half-formed ideas, links to related
tasks. Keep it terse.
